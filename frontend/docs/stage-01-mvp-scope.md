# Этап 1. Фиксация MVP и доменной модели

## Зачем нужен этот этап

До переноса макета в `frontend` нам нужен единый продуктовый фундамент. Без него следующие этапы быстро расползутся:

- макет будет переноситься как набор красивых экранов, а не как цельный сервис;
- FSD-сущности и фичи начнут проектироваться по-разному на каждом экране;
- интеграции `Docs <-> Tasks <-> Meetings <-> Releases` окажутся несогласованными;
- часть времени уйдет на вторичные сценарии, которые почти не влияют на оценку кейса.

Задача этапа: зафиксировать, что именно считаем hackathon MVP, какие сущности являются основными, как они связаны и в каком порядке эти связи должны появляться в продукте.

---

## Источники, на которые опираемся

- `figma/contracts/case.md` - бизнес-боль, критерии оценки, обязательные сценарии;
- `figma/contracts/openapi.yml` - целевой API-контур продукта;
- `figma/contracts/review.md` - нормализованная модель данных и связи;
- `figma/src/app` - интерактивный UI-референс целевых экранов;
- `frontend` - целевая кодовая база, куда переносится итоговый сервис.

---

## Продуктовая формулировка MVP

`Seamless` в рамках MVP - это единая операционная среда проекта, где вокруг проекта и эпохи собираются:

- задачи;
- документы и их версии;
- встречи и их итоги;
- релизы и pull request;
- уведомления;
- явные связи между всеми этими сущностями.

Главная ценность MVP: не просто показать отдельные разделы, а сократить ручную переклейку контекста между сервисами.

---

## Что обязательно входит в MVP

### 1. Docs-core

Обязательно должны быть:

- иерархия документов;
- области видимости документа;
- статус документа;
- версионирование документа;
- согласование изменений;
- связи документа с задачами, эпохами, встречами и релизами;
- inline-виджеты связанных сущностей внутри документа.

### 2. Kanban-core

Обязательно должны быть:

- канбан по задачам;
- карточка задачи как центральный узел связей;
- связь задачи с документами;
- связь задачи со встречами;
- связь задачи с PR и релизом;
- цитирование документа в обсуждение задачи.

### 3. Meetings

Обязательно должны быть:

- создание встречи из контекста задачи, документа или эпохи;
- участники встречи;
- согласование слотов;
- recap встречи;
- transcript / summary / decisions / action items;
- обратная привязка результатов встречи к документам и задачам.

### 4. CI/CD-core

Обязательно должны быть:

- release dashboard;
- PR dashboard;
- связь задачи с PR;
- отражение статуса PR в задаче;
- влияние релиза на эпоху и связанные сущности.

### 5. Общий интеграционный слой

Обязательно должны быть:

- проекция эпохи одновременно в задачи и документы;
- unified inbox;
- mentions / approval requests / meeting slot requests / PR events в одном месте;
- явное отображение графа связей в UX.

---

## Что не считаем обязательным для первого MVP

Это можно оставить на после-MVP или на уровень demo-эмуляции:

- реальная интеграция с внешним Git-провайдером;
- реальная видеоконференция и запись звонка;
- полноценный WYSIWYG-редактор корпоративного уровня;
- продвинутые workflow безопасности;
- автогенерация summary и transcript из реального медиа;
- полноценная auth-система с production-ready ролями и SSO.

На хакатоне важнее качественно показать модель, связи и пользовательские сценарии, чем довести внешние интеграции до промышленного состояния.

## Каноничные роли в MVP

### Customer

- видит только разрешенные проект и документы;
- участвует в согласовании документов и встреч;
- получает только релевантные уведомления;
- не должен видеть внутренние технические материалы.

### Developer

- работает с задачами, техническими документами, встречами, PR и релизами;
- видит внутренние связи сущностей;
- участвует в review и исполнении action items.

### Manager

- видит полную картину проекта;
- управляет эпохами, документами, встречами, релизами и связями;
- является главным бенефициаром unified UX.

## Главные связи, которые считаем обязательными

### Project как верхний контейнер

Каждый основной бизнес-объект должен быть видим в контексте проекта:

- `Project -> Epochs`
- `Project -> Tasks`
- `Project -> Documents`
- `Project -> Meetings`
- `Project -> Releases`
- `Project -> PullRequests`

### Epoch как сквозной спринт-контекст

Эпоха должна быть видна минимум в трех местах:

- в planning/workspace;
- в задачах;
- в документах.

Эпоха не должна быть изолированной сущностью. Пользователь должен видеть, какие документы, задачи, встречи и релиз относятся к текущему спринту.

### Task как центральный операционный узел

Задача должна связывать:

- документы;
- комментарии;
- встречи;
- PR;
- релиз;
- эпоху.

Именно карточка задачи является главным местом, где пользователь понимает, что нужно сделать, чего не хватает и что изменилось вокруг.

### Document как источник договоренностей

Документ должен иметь:

- статус;
- версии;
- владельцев;
- согласующих;
- комментарии;
- связи с задачами, встречами, эпохами и релизами.

Документ в MVP - это не вложение и не заметка, а first-class entity.

### Meeting как first-class entity

Встреча должна иметь:

- источник создания: `task` / `document` / `epoch`;
- участников;
- слоты и голоса;
- recap;
- transcript;
- summary;
- decisions;
- action items;
- связанные документы.

### Release / Pull Request как мост к разработке

Delivery-слой должен замыкать продуктовую историю:

- задача приводит к PR;
- PR влияет на статус задачи;
- PR входит в релиз;
- релиз влияет на готовность эпохи.

## Пять обязательных пользовательских сценариев MVP

### Сценарий 1. Docs -> Task

Пользователь открывает документ, видит связанный task status widget, вносит изменение и может процитировать фрагмент документа в обсуждение задачи.

Зачем:

- это прямо закрывает Docs-Kanban связность;
- это показывает, что контекст переносится без лишних переходов.

### Сценарий 2. Task -> Meeting

Из карточки задачи создается встреча, автоматически подтягиваются участники и контекст задачи, после встречи recap возвращается обратно в систему.

Зачем:

- это основной Kanban-core сценарий из кейса;
- он хорошо демонстрирует полезность сущности Meeting.

### Сценарий 3. Meeting -> Document

После встречи summary, decisions и update suggestions могут быть связаны с документом и повлиять на новую версию документа.

Зачем:

- это закрывает боль "почему была принята конкретная итерация";
- усиливает блок version history / approval.

### Сценарий 4. Task -> PR -> Release

Задача связана с PR, статус PR отражается в задаче, а релиз показывает готовность поставки.

Зачем:

- это обязательная часть интеграции CI/CD-core;
- без этого сервис выглядит как отдельный таск-трекер без связи с разработкой.

### Сценарий 5. Unified Inbox

Упоминания, approval requests, meeting slot requests и PR/release события приходят в единый inbox.

Зачем:

- это склеивает все остальные сценарии;
- это делает продукт единым, а не набором страниц.

## Приоритет реализации внутри MVP

Порядок не случайный, а завязан на критерии кейса и на архитектурную зависимость модулей друг от друга.

### P0. Обязательное ядро

- app shell;
- project context;
- epoch context;
- tasks;
- docs;
- docs-task связи.

### P1. Обязательная интеграция

- meetings;
- meeting recap;
- document versioning / approval;
- task-meeting-document flow.

### P2. Обязательная демонстрация связи с разработкой

- PR dashboard;
- release dashboard;
- task <-> PR <-> release связка.

### P3. Склейка продукта

- notifications;
- unified inbox;
- role-aware visibility;
- polishing сценариев.
